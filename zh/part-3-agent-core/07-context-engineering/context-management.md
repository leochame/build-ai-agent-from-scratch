# Context Management

Context Management 是 Agent 工程里最值得下力气的部分之一。

很多人一开始会把注意力放在模型能力、工具数量、提示词写法这些点上，但系统一旦变复杂，你很快就会发现，真正决定上限的往往不是“模型够不够强”，而是“给模型的上下文是不是对的”。

所以如果要用一句话概括 Context Management，我会把它理解成：

如何在有限的 token 预算里，让模型在当前时刻看到最值得看到的信息。

这里有两个重点。

第一，信息不是越多越好。  
上下文一旦过长，模型的注意力会被稀释，真正关键的信息可能反而埋掉。

第二，问题也不只是“塞多少”，而是“什么时候塞”“按什么结构塞”“哪些信息根本不该塞”。


## 1. 上下文管理到底在管理什么

先把一个边界讲清楚。

上下文不是记忆本身，上下文是模型在当前这一轮调用里真正能看到的内容。

比如：

- system prompt
- 用户当前输入
- 对话历史
- 工具调用结果
- 检索回来的资料
- 当前任务状态
- 临时生成的计划

这些内容一起，才构成这一次调用的 context。

而 memory 更像是一个信息仓库。它里面可以存很多东西，但只有当系统把某些记忆取出来，放进本轮调用里，它才会变成 context。

所以从工程角度看：

- memory 是潜在可用信息
- context 是当前实际注入信息

Context Management 管理的，就是从所有潜在信息里，挑出当前最值得给模型看的那一部分。

## 2. 我会怎么拆这个问题

如果从上下文真正形成的过程来看，我会把 Context Management 大致分成两部分：

- pre-handle
- post-handle

这个划分不一定是标准术语，但它在工程上是很有用的。

`pre-handle` 关注的是：在模型调用之前，系统怎样决定“该拿什么信息进来”。

`post-handle` 关注的是：在模型调用之后，系统怎样处理已经产生的上下文和结果，避免上下文不断膨胀、污染和失控。



## 3. Pre-handle

目前来看，pre-handle 的核心思想其实就是 `Progressive Disclosure`。

也就是不要一开始把所有信息都摊给模型，而是按需要逐步暴露。

这个思路看起来很朴素，但它背后对应的是一个很重要的工程判断：

Agent 不是在“拿到全部信息后再思考”，而是在“有限信息下，逐步决定下一步还需要什么”。

这和很多传统 prompt 设计不太一样。  
传统做法经常是尽可能把背景一次性写全，但在复杂系统里，这种方式很快就会遇到几个问题：

- token 成本过高
- 大量信息与当前任务无关
- 关键线索被埋没
- 系统越来越难维护

所以 pre-handle 的目标不是“信息尽量全”，而是“信息尽量准”。

### 3.1 Skill

`skill` 是这几年很火的一种上下文注入方式。

虽然很多人会觉得它有营销成分，但如果把概念拆开来看，它真正重要的地方不在名字，而在它体现出来的思路：

把一组只在特定场景下才需要的知识、流程、约束和操作习惯，延迟到真正需要的时候再注入。

这本质上就是 Progressive Disclosure。

一个 skill 里通常会包含这些东西：

- 某类任务的操作说明
- 对应任务的最佳实践
- 需要注意的边界条件
- 常见错误和规避方式
- 可用工具说明

它的价值不在于“多了一个文件”，而在于它把原本不应该始终驻留在主 prompt 里的内容，从默认上下文里拆了出去。

至于实现方式，其实很多样：

- 用本地文件组织 skill
- 用检索系统按需召回 skill
- 用 function calling 触发 skill 加载
- 用目录搜索、关键词匹配、grep 等方式选择相关说明

所以 skill 不一定是一种单独技术，更像是一种上下文组织策略。

### 3.2 Tool Search

当 Agent 可用工具越来越多时，一个非常现实的问题就出现了：

模型到底该看到全部工具，还是只看到当前可能相关的工具？

如果工具很少，全部暴露问题不大。  
但一旦工具多起来，工具描述本身就会变成巨大的上下文负担。

这时候就需要 `Tool Search`。

它的核心不是让模型“记住所有工具”，而是先做一层工具筛选，再把候选工具暴露给模型。

这里通常会做几件事：

- 根据任务类型筛选工具
- 根据用户意图匹配工具
- 根据当前状态排除不该用的工具
- 根据权限或环境限制过滤工具

这样做有几个直接收益：

- 降低 prompt 长度
- 减少模型误用工具的概率
- 提高工具选择的稳定性
- 让工具体系更容易扩展

从这个角度看，Tool Search 其实就是“工具层面的上下文管理”。

### 3.3 Code Act

在 coding agent 里，`Code Act` 也是很典型的 pre-handle 问题。

因为代码库本身就是一个巨大的上下文池，不可能每次都完整塞给模型。

所以系统必须先回答几个问题：

- 当前任务最相关的是哪些文件
- 应该先读哪些入口文件
- 哪些符号、函数、模块值得继续展开
- 是不是要先搜索，再阅读，再编辑

这其实就是把“代码理解过程”变成一种上下文选择过程。

常见做法包括：

- 先用文件名、目录结构做粗筛
- 再用关键词、符号搜索做精筛
- 只展开与当前任务最相关的代码片段
- 把长文件拆成局部阅读，而不是一次性全读

如果这个过程做得不好，模型就会有两种典型问题：

- 看得太少，理解不够
- 看得太多，注意力分散

所以 Code Act 的本质也不是“让模型读代码”，而是“让模型按任务需要逐步看到代码”。

## 4. Post-handle

如果说 pre-handle 决定“上下文怎么来”，那么 post-handle 决定“上下文怎么收”。

这部分同样重要，因为 Agent 一旦进入多轮交互、频繁工具调用、长任务执行之后，上下文会天然膨胀。

如果没有 post-handle，系统很容易出现这些问题：

- 历史信息不断堆积
- 低价值内容长期占据窗口
- 工具返回结果重复注入
- 旧计划污染新任务
- token 成本持续走高

所以 post-handle 的目标是：

让上下文保持连续性，但不要失控。

### 4.1 Context Compression

最常见的一类 post-handle 手段就是压缩。

压缩的重点不是机械缩短，而是保留对后续推理真正有价值的信息。

比如一轮长对话结束后，真正值得保留的可能不是每一句原话，而是：

- 当前任务目标
- 已经做过的关键决策
- 还没解决的问题
- 工具调用得到的关键事实
- 下一步计划

也就是说，压缩不是简单摘要，而是面向后续行动的信息重组。

### 4.2 Context Unloading

另一类重要手段是卸载。

有些信息在某一阶段很重要，但在后续阶段已经没有必要继续留在主上下文里。

比如：

- 已完成子任务的中间日志
- 大段工具原始输出
- 已确认无关的检索结果
- 已经被总结进状态里的对话细节

这些内容不一定要删除，但可以从主上下文中移走，只在需要时再取回。

这其实就是把“长期保留”和“当前暴露”分开。

### 4.3 State Extraction

post-handle 还有一类很重要但容易被忽视的工作，就是把交互过程提炼成结构化状态。

例如：

- 当前任务阶段
- 已完成步骤
- 失败原因
- 用户偏好
- 待确认事项

如果这些信息一直只存在于自然语言对话里，那么系统后面每一步都要重新理解一遍。  
但如果它们被抽成结构化 state，后续上下文装配就会轻很多。

所以很多时候，post-handle 做得好，实际上也会反过来提升 pre-handle 的质量。

## 5. 一个更实用的理解方式

如果不想把 Context Management 理解得太抽象，我觉得可以直接把它看成三个连续问题：

1. 什么信息值得进入当前上下文？
2. 这些信息应该以什么结构进入？
3. 这轮结束后，哪些内容该保留，哪些该压缩，哪些该卸载？

只要这三个问题没有想清楚，Agent 系统就很容易陷入一种表面上能跑、实际上很脆弱的状态。

## 6. 为什么这部分这么重要

我现在越来越倾向于认为，很多 Agent 的工程问题，最后都会落回 Context Management。

比如：

- 为什么模型忽然变笨了
- 为什么工具选择不稳定
- 为什么长任务越做越乱
- 为什么 token 成本越来越高
- 为什么系统一扩展就开始失控

这些问题表面看起来分散，但很多时候根源都不是模型本身，而是上下文组织出了问题。

所以如果只把 Context Management 理解成“截断历史消息”或者“做个摘要”，其实是远远不够的。

它真正对应的是一整套问题：

- 信息如何选择
- 信息如何分层
- 信息如何延迟暴露
- 信息如何沉淀为状态
- 信息如何在多轮过程中持续保持干净

## 7. 最后

如果说 Prompt Engineering 关注的是“怎么把一句话写好”，那么 Context Management 更关注的是“怎么把一个系统长期喂好”。

它不只是一个 prompt 技巧，而是 Agent 架构能力的一部分。

在我看来，这部分甚至比很多花哨的 Agent 概念更值得反复琢磨。因为一个系统是否真的能走向复杂任务，最后很大程度上取决于它有没有能力持续构造高质量上下文。

# 前言：这不是一套教程，而是我的自学路程

如果你打开了这个仓库，我想先说明一件事：

这不是一套已经设计完整、结构严密、按教学目标编排好的标准教程。

它更像是我在学习 AI Agent 这件事上的一条自学路径，一份持续整理的笔记，一张正在慢慢画清楚的地图。

我会把我理解过的东西、踩过的坑、卡住过的问题、重新想明白的过程，都放进这个仓库里。

所以这里的内容，不应该被理解为“我来教你怎么学”，而更接近于：

- 我是怎么开始理解 AI Agent 的
- 我在哪些地方反复困惑过
- 我为什么会把内容拆成现在这样的结构
- 我目前认为哪些问题重要，哪些问题其实被说得太热闹了

如果这些记录刚好也能帮到你，那当然很好。

## 为什么我想把这段路程记下来

开始学 AI Agent 之后，我很快就有一种很熟悉的感觉：

- 资料很多
- 新名词很多
- 框架很多
- Demo 也很多

但真正把这些东西连起来的时候，脑子里常常还是乱的。

有时候你以为自己已经懂了，结果一到自己动手搭，就会发现很多问题根本没有想清楚：

- Agent 和普通聊天机器人到底差在哪
- 模型到底擅长什么，不擅长什么
- Prompt、message、context 这些词到底是不是一回事
- 工具调用到底是“模型会了”，还是“系统设计对了”
- RAG、memory、可靠性这些概念之间到底怎么分工
- Multi-Agent 到底是能力升级，还是复杂度升级

我后来慢慢意识到，自己缺的不是再看一篇“十分钟上手 Agent”的文章，而是一条更连续的学习路径。

于是这个仓库就出现了。

它的目的不是把我包装成一个“在讲课的人”，而是把我自己的理解过程留下来。

## 这份仓库更像什么

如果一定要给它一个定位，我觉得它更像是下面三样东西的结合：

- 一份自学过程中的结构化笔记
- 一份我给自己整理的 Agent 知识地图
- 一个会随着理解变化而不断被重写的项目仓库

这意味着几件事：

第一，它不一定会一开始就很完整。  
有些章节可能很长，有些章节可能只有一个开头，有些地方的理解以后也可能会改。

第二，它不追求“像教程一样面面俱到”。  
我会优先写那些我自己真的卡过、想过、反复比较过的问题。

第三，它会带有很强的个人路径。  
也就是说，这不是“唯一正确的学习顺序”，而是“我目前觉得比较自然的一条路”。

## 为什么我不想把它写成“教程”

因为“教程”这个词，很容易给人一种错觉：

- 结构应该已经完全定好了
- 每一章都应该有明确结论
- 作者应该站在一个已经完全掌握全局的位置上

但我现在做的并不是这件事。

我更接近于一边学，一边整理，一边重构自己的理解。

有些内容我已经想得比较清楚了，有些内容还在继续摸索。

所以如果我把它写成“教程”，反而会掩盖这个仓库最真实的价值：

它记录的不是一个已经完成的答案，而是一个不断逼近答案的过程。

我觉得这反而更诚实。

## 我目前是怎么理解这条学习路径的

虽然这不是标准教程，但它也不是完全随手乱记。

我还是希望它有一条尽量清晰的主线。所以目前我把这条自学路径大致拆成了几个阶段。

### 第一段：先建立基本地图

先不要急着追框架，也不要急着堆功能。

先想清楚几件基础问题：

- AI Agent 到底是什么
- 它和 Chatbot 的差异到底在哪里
- LLM 在这个系统里到底扮演什么角色
- 为什么上下文会成为几乎所有问题的核心

这一段更像是在画地图。

地图不一定能立刻帮你把东西做出来，但没有地图，你后面很容易一路都在局部打转。

### 第二段：把最小闭环搭起来

等基础地图稍微有了之后，再去看最小可用 Agent 需要什么。

比如：

- 基础对话
- message 组织
- prompt 设计
- state 管理
- function calling
- 工具接入

这一段很重要，因为很多“高级问题”，最后都能往回追溯到这些最基础的结构有没有搭稳。

### 第三段：开始理解真正决定上限的东西

学到这里之后，我越来越强烈地感觉到：

很多 Agent 问题，本质上不是“模型不够强”，而是“上下文没有组织好”。

所以后面的重点自然会落到这些主题上：

- context engineering
- context management
- memory
- RAG
- evaluation

这些内容通常没有“跑个 demo 就懂了”那么轻松，但它们更接近真实工程里最难、也最关键的部分。

### 第四段：再去看更复杂的系统形态

比如 Multi-Agent。

这个话题当然很重要，也很吸引人，但我现在越来越不想把它当作起点。

因为如果单 Agent 的边界、上下文、工具、执行流程都还没想清楚，那么多 Agent 往往只是把问题复制成更多份。

所以对我来说，Multi-Agent 更像是后面的延伸问题，而不是一开始就该冲进去的方向。

## 关于框架，我现在的态度

我不会把这个仓库写成某个框架的使用手册。

不是因为框架不重要，而是因为我现在更想先弄明白：

- 这些抽象到底解决了什么问题
- 如果不用框架，底层链路应该怎么理解
- 哪些能力是框架帮你封装了，哪些代价也被一起藏起来了

只有先把这些想清楚，后面再看框架时，心里才不会那么虚。

否则很容易变成：

- 会调接口
- 会调用框架 API
- 会跑示例

但一旦要自己改、自己搭、自己排查问题，就开始发虚。

## 这份仓库会有什么样的内容

这里不会只有“结论”。

我也会尽量保留这些东西：

- 一些我原来以为懂了，后来发现没懂的地方
- 一些概念之间真正容易混淆的边界
- 一些从 Demo 走向工程实现时才会暴露的问题
- 一些我自己还没有完全想明白，但值得继续追的问题

所以这份仓库大概率会显得不那么“标准”，但会更接近真实学习过程。

## 如果你也在学，可以怎么使用这个仓库

我会更建议你把它当成一个同行者的笔记，而不是一本权威教程。

你可以用它来做这些事：

- 看我现在是怎么组织这张知识地图的
- 对照自己目前卡在哪一层
- 找到下一步应该补的主题
- 顺着章节继续往下读，也顺便保留自己的判断

如果你发现某些地方你有不同理解，那也完全正常。

这本来就不是一个“标准答案仓库”，而是一条正在展开的学习路径。

恰恰相反，是因为我还在学，所以我想把这条路留下来。

我希望以后回头看的时候，能看见自己是怎么一步一步把这些概念、问题和工程实践串起来的。

如果你也刚好走在类似的路上，希望这份记录能给你一点参考。

不一定是答案，但至少是一条真实走过的路径。
